# Технический Отчет Проекта CardVault

Этот документ представляет собой исчерпывающий технический отчет по архитектуре, структурам данных и фронтенд-логике e-commerce платформы для продажи коллекционных и кастомных карточек.

## 1. ТЕХНОЛОГИЧЕСКИЙ СТЕК И АРХИТЕКТУРА

*   **Frontend:** Приложение построено на **React 18** с использованием фреймворка **Next.js 14.2 (App Router)**. В качестве подхода к рендерингу используется гибридная модель: SSR для начальной загрузки и SEO, а также клиентские компоненты (`"use client"`) для интерактивных элементов (каталог, корзина, фильтры).
*   **Backend:** Роль бэкенда выполняют **Next.js API Routes** (Serverless Functions) поверх **Node.js 22**.
*   **База данных и ORM:** Используется **Firebase Firestore** (NoSQL, Modular SDK v10+). ORM или классического Query Builder нет — запросы выполняются напрямую через Firestore SDK (`collection`, `getDocs`, `where`).
*   **Хранение статики и медиа:**
    *   Собственные изображения приложения хранятся в папке `public/`.
    *   Пользовательские медиа и медиа-архив карточек загружаются в **Cloudflare R2** (S3-совместимое объектное хранилище). Взаимодействие происходит через `@aws-sdk/client-s3`.
    *   *Критично:* На бэкенде (в `app/api/upload/route.ts`) используется библиотека `sharp` для **обязательной конвертации всех загружаемых изображений в формат WebP** с оптимизацией перед отправкой в S3/R2.
*   **Уведомления/Интеграции:** Приложение использует **Telegram Bot API** (в `app/api/send-order/route.ts`) для обработки и отправки уведомлений о заказах в специальный канал, что заменяет традиционные очереди и e-mail рассылки на данном этапе.

## 2. СТРУКТУРА БАЗЫ ДАННЫХ И МОДЕЛИ (Data Models)

Так как используется Firestore, жестких реляционных схем нет, но структура данных задается через TypeScript интерфейсы и Zod-валидацию в компонентах админки.

### Card (Карточка)
Метаданные хранятся в виде плоского документа в коллекции `cards`. EAV-паттерн не используется; структура фиксированная:
```typescript
interface Card {
  id: string;                    // Генерируется Firestore
  name: string;                  // Системное имя (обычно = title)
  title: string;                 // Отображаемое название
  price: number;                 // Текущая цена
  originalPrice?: number;        // Старая цена (для скидок)
  discount?: number;             // % скидки
  rarity: "common" | "rare" | "epic" | "legendary"; // Редкость
  image: string;                 // URL основного изображения (legacy-поле)
  imageUrl: string;              // URL основного изображения (WebP из R2)
  bannerImageUrl?: string;       // URL баннера (если карточка продвигается)
  cardBackImageUrl?: string;     // URL рубашки/задника карточки
  carouselImageUrl?: string;     // URL для карусели
  isFeatured: boolean;           // Флаг вывода на главную
  category: string;              // Категория (название текстом)
  description: string;           // Текстовое описание
  inStock: boolean;              // Остаток на складе (boolean, без счетчика кол-ва)
  isHot: boolean;                // Флаг "Хит продаж"
}
```
*Валидация при создании/редактировании обеспечивается библиотекой Zod (`cardSchema` в `components/admin/card-form.tsx`).*

### CustomOrder (Кастомный заказ) / Order & Payment
Полноценная сущность "Заказ" в БД (со статусами "Оплачен", "В доставке") не обнаружена. Весь процесс сводится к формированию заявки (массив товаров из корзины + контакты) и отправке её через Telegram API. Платежные шлюзы (Stripe/Robokassa) на текущий момент не подключены.
```typescript
interface OrderData {
  name: string;      // Имя покупателя
  phone: string;     // Телефон
  items: Array<{
    title: string;
    quantity: number;
    price: number;
  }>;
  total: number;     // Итого к оплате
  discount: number;  // Сумма скидки
}
```

### User & Auth
Используется **Firebase Auth**. Ролевая модель (Admin/User) управляется, скорее всего, через Custom Claims во Firebase или отдельную коллекцию (точный механизм скрыт в `app/admin` логине).

Корзина пользователя **не хранится в базе данных**. Она управляется исключительно локально через состояние React контекста / Zustand (см. `lib/cart-context.tsx`) и, вероятно, синхронизируется с `localStorage`.

## 3. БЭКЕНД И API

*   **Архитектура API:** Гибридная. Данные каталога фетчатся клиентом напрямую из Firestore. Кастомные операции (загрузка файлов, отправка заказов) реализованы через RESTful Next.js API Routes. Swagger/OpenAPI нет.
*   **Аутентификация:** Firebase Auth (вероятно JWT-токены). Защита роутов (например, админки) реализована либо через middleware Next.js, либо через проверку сессии в корневом layout/page админки.
*   **Пагинация и фильтрация:**
    *   *Проблема:* Пагинация в каталоге (`components/catalog-grid.tsx`) заявлена в UI (переменные `currentPage`, `itemsPerPage`), но **не реализована на уровне запросов в БД**.
    *   Метод `getDocs(collection(db, "cards"))` вытягивает **всю коллекцию карточек сразу**. Полнотекстового поиска (Elasticsearch/Meilisearch) нет. Сортировка и фильтрация применяются к массиву данных уже на клиенте (in-memory).
*   **Обработка медиа:**
    *   Реализован флоу `multipart/form-data` через роут `/api/upload`.
    *   Сервер принимает файл, проверяет тип, и если это не WebP — конвертирует его через `sharp` (`quality: 85`, `effort: 4`).
    *   Имена файлов санируются `sanitizeFileName` для защиты от path traversal.
    *   Затем файл пушится в Cloudflare R2 через AWS SDK `PutObjectCommand`. Клиенту возвращается публичный URL `R2_PUBLIC_URL`.

## 4. FRONTEND И UX-ЛОГИКА

*   **State Management:**
    *   Состояние корзины управляется через глобальный стейт (Контекст или `zustand`, см. `lib/cart-context.tsx`).
    *   Состояние фильтров (`priceRange`, `categories`) — локальный стейт React `useState` в компоненте `CatalogFilters`.
*   **UI/UX компоненты:**
    *   Стилизация: **Tailwind CSS**.
    *   UI-библиотека: **shadcn/ui** (построена поверх **Radix UI** примитивов). Используются `Card`, `Select`, `Button`, `Slider`, `Badge`.
    *   Анимации: **Framer Motion** (`framer-motion`) и кастомные CSS/Tailwind анимации (`tailwindcss-animate`).
*   **Custom Builder:**
    *   В текущем анализе кода не найдено сложных компонентов визуального конструктора (canvas, drag-n-drop). Создание кастомных карточек, вероятно, реализовано через стандартные формы загрузки изображений, но не через интерактивный canvas-редактор.
*   **Дизайн система:**
    *   Темная тема по умолчанию (`bg-zinc-950`).
    *   Карточки товаров используют прозрачные фоны (`bg-transparent`) без рамок, изображения позиционируются с использованием `drop-shadow` для эффекта "парения".

### Инфраструктура
*   **CI/CD & Docker:** Файлы `Dockerfile` или `docker-compose.yml` в корне отсутствуют. Деплой, вероятнее всего, происходит напрямую на Vercel (стандартный флоу Next.js проектов), о чем косвенно свидетельствует зависимость `@vercel/analytics`.
*   **Сторонние сервисы:** Firebase (Auth, Firestore, Storage), Cloudflare R2 (Медиа), Telegram (Нотификации), Vercel Analytics. Платежные агрегаторы отсутствуют.

### Технический долг (Technical Debt) & Проблемы
1.  **Отсутствие пагинации в БД (N+1 / Оver-fetching):**
    В компоненте каталога весь инвентарь скачивается разом (`getDocs(collection(db, "cards"))`). При росте базы (тысячи карточек) это приведет к серьезным просадкам производительности, избыточному потреблению памяти на клиенте и высоким костам (Firestore тарифицирует чтение каждого документа). Нужно внедрять серверную пагинацию (limit, startAfter).
2.  **Зависимость корзины от Client-State:**
    Поскольку корзина живет только на клиенте, пользователи теряют собранные заказы при смене устройства (кросс-девайс сессии не работают).
3.  **Отсутствие серверной валидации при отправке заказа:**
    API роут `app/api/send-order` не использует Zod (или аналог) для валидации структуры `req.body`. В `POST` обработчике происходит `await request.json()` с простым кастом к интерфейсу TypeScript, что оставляет уязвимость: злоумышленник может отправить некорректные или вредоносные данные. Скидка и финальная цена приходят с клиента, что позволяет подделать сумму платежа. Пересчет `total` должен происходить строго на сервере по ID товаров.
4.  **Отсутствие системы платежей и трекинга статусов заказов:**
    Заказ отправляется в Telegram и "забывается" системой. Нет возможности клиенту проверить статус заказа или получить чек об оплате.
5.  **Бизнес-логика в компонентах:**
    В компоненте формы создания карточки (`components/admin/card-form.tsx`) зашиты сложные правила пересчета `price`, `originalPrice` и `discount`. Это лучше выносить в отдельные utility-функции для покрытия тестами.
